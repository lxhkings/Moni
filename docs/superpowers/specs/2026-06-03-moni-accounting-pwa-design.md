# Moni 记账 PWA — 设计文档

日期：2026-06-03
状态：已确认，待实现规划

## 1. 背景与目标

鲨鱼记账广告过多。需要一个自用 / 家人用的替代品：无广告、隐私优先、数据本地。
需支持安卓与 iOS 双端，并能在换手机时通过扫二维码迁移记录。

**定位**：自用 / 家人用，不上架应用商店，无后端，无账号，无广告。

**非目标（明确不做）**：
- 不上架 App Store / 应用商店
- 不做云同步、不做用户账号体系
- 不做多账本、不做资产管理 / 多账户对账
- 不做商业化（订阅 / 捐赠）

## 2. 法律说明（决策依据）

做同类记账 App 合法；功能创意不受版权保护。本项目为自用、不公开发行，法律风险基本消失。
即便未来公开，安全线为：自起名（Moni）、自设计 UI、自写代码，只借鉴"功能集合"；
不碰鲨鱼记账的名字 / Logo / 素材 / 代码。

## 3. 技术选型与理由

| 项 | 选择 | 理由 |
|---|---|---|
| 形态 | **PWA（渐进式网页 App）** | 绕开 iOS 签名机制：免费 Apple ID 签名的原生 App 每 7 天过期，PWA 走浏览器"加到主屏"，不过期、零编译、零费用、双端一套代码 |
| 框架 | **Vue 3 + Vite** | 轻量、PWA 模板成熟、上手快（React 可替换，无强制） |
| 存储 | **IndexedDB**（封装 Dexie.js） | 结构化、容量大、本地持久 |
| 图表 | **ECharts** 或 Chart.js | 饼图 + 折线够用 |
| 二维码 | **qrcode**（生成）+ **jsQR**（扫描，调摄像头） | 迁移功能 |
| PWA 工具 | **vite-plugin-pwa** | 自动生成 Service Worker + manifest |
| 托管 | **免费静态托管**（Vercel / Netlify / GitHub Pages） | 免费、自带 HTTPS（PWA 必需）、一键部署 |

**关键约束**：iOS 强制代码签名，免费证书 7 天失效；PWA 走浏览器规避此约束，故"不过期"。
此为选择 PWA 而非 Flutter / React Native 原生方案的核心原因。

## 4. 架构

```
纯前端 PWA（无后端、无账号）
┌─────────────────────────────┐
│  UI 层 (记账 / 报表 / 预算 / 设置) │
├─────────────────────────────┤
│  状态层 (内存 + 响应式)            │
├─────────────────────────────┤
│  存储层 (IndexedDB / Dexie 本地)  │
├─────────────────────────────┤
│  Service Worker (离线缓存)        │
└─────────────────────────────┘
迁移：旧机导出 → 二维码 → 新机扫码导入
托管：免费静态托管 (HTTPS)
```

数据永不离开手机：无隐私问题、无广告、无网络依赖（首次加载后离线可用）。

## 5. 数据模型（单账本）

| 表 | 字段 |
|---|---|
| `transactions` | id, type(income/expense), amount, categoryId, note, date, createdAt |
| `categories` | id, name, icon, type(income/expense), sortOrder |
| `budgets` | id, month(YYYY-MM), categoryId(空=总预算), amount |
| `meta` | schemaVersion, currencySymbol |

- 金额以最小货币单位（分）整数存储，避免浮点误差。
- 首次启动写入一套默认分类（餐饮、交通、购物、居住、工资等）。
- `schemaVersion` 用于未来迁移与导入兼容校验。

## 6. 界面（底部 4 Tab）

| Tab | 内容 |
|---|---|
| **明细** | 按日分组的流水列表，顶部显示本月收入 / 支出 / 结余汇总；点条目可编辑 / 删除 |
| **记一笔** | 大数字键盘输金额 → 选分类（图标网格）→ 选日期 / 备注 → 保存（核心高频流程） |
| **报表** | 月 / 年切换；分类支出饼图；收支趋势折线图 |
| **我的** | 预算设置、分类管理、数据导出 / 扫码迁移、货币符号设置 |

## 7. 二维码迁移

**导出（旧机）**：
1. 读取全部数据（transactions + categories + budgets + meta）→ JSON
2. gzip 压缩 → Base64 编码
3. 数据小（≤ ~2KB）：生成单张二维码
4. 数据大：分页切片，**多张二维码轮播展示**，每张带序号 / 总数标记

**导入（新机）**：
1. 调摄像头连续扫码，按序号拼接所有分片
2. 集齐后 Base64 解码 → 解压 → JSON
3. 校验 `schemaVersion` → 选择**覆盖**或**合并**导入

**兜底备份**：额外提供 **JSON 文件导出 / 导入**，防 iOS 在存储紧张时清理 IndexedDB。

全流程本地，无服务器参与。

## 8. 测试策略

**单元测试**：
- 数据层增删改查
- 金额 / 预算计算（含分→元换算）
- 导出 → 导入往返数据一致性（round-trip）
- 二维码分片切割 / 拼接还原

**手动验证**：
- 双机实走一遍扫码迁移（小数据 + 大数据两种）
- iOS Safari "加到主屏" 后断网离线可用
- 安卓 Chrome 同上
